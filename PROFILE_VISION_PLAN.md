## Главные Приоритеты Пользователя

Ниже не просто backlog. Это порядок, в котором профиль должен считаться "правильным".

### P0. Приватность И Прямой Доступ
1. `/профиль` должен открывать профиль сразу, без лишней промежуточной логики.
2. Сообщение `профиль` должно работать как естественная текстовая точка входа.
3. `@юзер профиль` и reply-сценарий должны открывать профиль нужного игрока.
4. Сам профиль должен быть виден только инициатору.

### P0. Ограничения Доступа
1. Все `dead`-участники не должны иметь доступа к профилю вообще.
2. Участник без уникального server tag не должен открывать чужие профили.
3. Staff/moderator bypass обязан работать везде одинаково.

### P0. Один Экран, Не Куча Сообщений
1. Профиль должен быть собран в одном окне.
2. Переключение разделов должно жить внутри одного Components V2 payload.
3. Интерфейс должен ощущаться как единый приватный workspace по игроку, а не как набор командных ответов.

### P1. Богатство Данных, Которые Вы Явно Просили
1. Статистика по серверу.
2. Роли.
3. Мейны.
4. Wiki/guide-связки.
5. Последние изменения по kills, в подробностях.
6. Roblox-linked account.
7. Ранги.
8. ELO.
9. Tierlist.
10. Кнопки переходов и полезных действий.
11. Аватары и медиа-подача.

### P1. Хорошая Архитектура
1. Не запутано.
2. Стабильно.
3. Без повторения одной и той же логики в разных ветках.
4. С чётким owner per behavior.

## Целевое Продуктовое Видение

Профиль должен быть не "карточкой ради карточки", а приватным центром игрока внутри сервера.

Целевая форма профиля:
1. Открывается быстро и естественно из slash или текста.
2. Показывает короткую верхнюю сводку сразу.
3. Даёт внутри того же окна переход по отдельным разделам аналитики.
4. Собирает не только identity, но и progression, activity, ranking и social/Roblox контекст.
5. Не раскрывает чужие данные в публичный канал.
6. Не разваливает кодовую базу и не размазывает логику по `welcome-bot.js`.

## Архитектурное Видение

### Канонические Owner-Seams
1. [src/profile/access.js](src/profile/access.js)
   - владелец policy и deny-reasons,
   - ничего не знает о Discord UI.
2. [src/profile/entry.js](src/profile/entry.js)
   - владелец parsing для текстового trigger,
   - владелец `profile_open` / `profile_nav` custom ids.
3. [src/profile/model.js](src/profile/model.js)
   - владелец read-model,
   - собирает данные из shared profile, Roblox, ELO, tierlist, submissions.
4. [src/profile/view.js](src/profile/view.js)
   - владелец Components V2 payload,
   - владелец секций, navigation row и helper-message payload.
5. [src/profile/operator.js](src/profile/operator.js)
   - владелец runtime orchestration,
   - владелец slash/message/button profile handlers,
   - единственная точка, где сводятся access + read-model + render.
6. [welcome-bot.js](welcome-bot.js)
   - только thin router,
   - не должен снова становиться владельцем profile behavior.

### Что Это Даёт
1. Профиль можно развивать без повторения одной и той же логики в четырёх Discord entrypoints.
2. Pure logic можно тестировать отдельно от Discord runtime.
3. UI можно усиливать без переписывания ACL и parsing.
4. Будущие расширения по Roblox/ranks/wiki не обязаны лезть в hot path напрямую.

## Подробный План По Приоритетам

## Фаза 1. Закрыть Продуктовый V1 На Живом Сервере

Это самый высокий приоритет после уже написанного кода, потому что вы просили не просто красивую архитектуру, а рабочий профиль.

### 1.1. Discord Smoke По Базовым Сценариям
Нужно руками проверить:
1. `/профиль` открывает свой профиль.
2. `/профиль target:@user` открывает чужой профиль при наличии tag.
3. `профиль` отправляет helper-message и открывает приватный payload.
4. `reply + профиль` берёт автора reply как target.
5. `@user профиль` корректно берёт target по mention.
6. `profile_open` button открывает правильный профиль и удаляет helper-message.
7. `profile_nav` кнопки обновляют тот же приватный payload.

### 1.2. Smoke По Access Policy
Нужно руками проверить:
1. Пользователь в `dead` не открывает даже свой профиль.
2. Пользователь без tag открывает только свой профиль.
3. Пользователь без tag не открывает чужой профиль.
4. Staff/mod bypass обходит оба ограничения.

### 1.3. Проверка Реальных Данных
Нужно проверить на живых профилях:
1. roles совпадают с Discord member state,
2. mains совпадают с shared profile / role-derived main view,
3. kills/progression не расходятся с latest approved data,
4. ELO и tierlist не тянут устаревшие поля,
5. Roblox block не показывает мусор при частично заполненном состоянии.

## Фаза 2. Довести Профиль До Вашего Запрошенного Богатого Содержания

Это главный продуктовый этап после smoke, потому что именно здесь закрываются ваши слова про "круче", "подробнее" и "по системе как на скрине".

### 2.1. Верхняя Сводка
Нужно усилить верхний overview block так, чтобы он сразу давал:
1. Discord display name,
2. target mention,
3. ключевые роли,
4. mains,
5. current progression markers,
6. Roblox identity summary,
7. статус доступа или важные flags, если они нужны.

### 2.2. Раздел Серверной Аналитики
Нужно собрать отдельный богато объяснённый блок по activity/server stats:
1. сообщения 7/30/90 дней,
2. сессии 7/30/90 дней,
3. активные дни,
4. дни на сервере,
5. last seen,
6. activity bucket,
7. статус eligibility, если он важен для роли/доступа.

### 2.3. Раздел Прогресса И Боевого Ростa
Это один из самых важных слоёв, потому что вы явно просили kills и изменения по ним.

Нужно довести до отдельной сильной секции:
1. current approved kills,
2. tier / rank,
3. место среди проверенных игроков,
4. доля от общих серверных kills,
5. последняя approved проверка,
6. последний рост по kills,
7. темп роста за период,
8. при возможности маленькую историю нескольких последних изменений, а не только одну строку summary.

### 2.4. Раздел ELO И Tierlist
Нужно свести ranking-сигналы в один понятный раздел:
1. current ELO,
2. current ELO tier,
3. статус последней ELO submission,
4. tierlist main,
5. influence/lock/status,
6. дополнительные производные статусы, если они реально полезны игроку.

### 2.5. Раздел Roblox И Social Context
Нужно усилить Roblox/social block до полноценного identity-context слоя:
1. Roblox username,
2. verified binding status,
3. Roblox profile link,
4. server friends count,
5. JJS playtime / minutes,
6. frequent non-friend signals,
7. last seen in JJS,
8. при наличии данных — Roblox avatar/media.

### 2.6. Мейны, Wiki И Guide-Связки
Нужно довести именно вашу связку mains + wiki/guides:
1. main list должен быть заметным блоком, а не второстепенной строкой,
2. combo/guide links уже есть и должны стать оформленным action block,
3. если появится wiki source, его нужно добавить как отдельный canonical link layer,
4. по каждому main в идеале нужен быстрый переход в guide/wiki.

### 2.7. Аватары И Media Presentation
Это отдельный пользовательский приоритет, базовый слой которого уже реализован, но ещё не отполирован.

Что уже принято и реализовано:
1. Discord avatar используется как primary visual identity, когда доступен,
2. Roblox avatar добавляется как дополнительное media,
3. Components V2 media/gallery block уже используется,
4. media не должен дублировать один и тот же источник несколько раз.

Что ещё довести:
1. визуальный баланс блока,
2. правила приоритета между Discord/OAuth/Roblox avatar в спорных кейсах,
3. поведение на partial-data профилях,
4. финальную подачу без перегруза интерфейса.

Рекомендуемая цель:
1. верхний visual identity block,
2. link/media buttons рядом,
3. без дублирования и без превращения профиля в баннерный мусор.

## Фаза 3. Полировка UX Внутри Одного Окна

### 3.1. Навигация
1. Переключение секций должно быть мгновенным и предсказуемым.
2. Названия секций должны отражать реальный смысл, а не только технический layout.
3. Нужно проверить, какие секции реально нужны игроку в первом экране, а какие вторичны.

### 3.2. Empty States И Partial States
1. Если данных нет, профиль должен объяснять это ясно.
2. Если Roblox или ranking данные не заполнены, интерфейс не должен выглядеть сломанным.
3. Если часть данных есть, а часть нет, нужно показывать честные partial states без выдумок.

### 3.3. Язык И Формулировки
1. Профиль должен говорить по-русски последовательно.
2. Нужны более сильные и читаемые подписи вместо сухих технических строк там, где это важно игроку.
3. При этом нельзя превращать UI в художественный текст с потерей точности.

## Фаза 4. Архитектурная Доводка После Продуктового V1

Это не выше ваших продуктовых требований, но важно для долгой жизни кода.

### 4.1. Свести Старые Legacy Surface-ы
1. Найти старые profile/my-card ветки в [welcome-bot.js](welcome-bot.js) и соседних legacy поверхностях.
2. Решить, какие из них удалить, какие адаптировать, а какие временно оставить как bridge.
3. Не допустить появления второй независимой профильной системы.

### 4.2. Довести Operator До Полного Канонического Владения
Сейчас уже сделан основной шаг. Дальше важно не откатиться обратно.

Нужно удержать правило:
1. новые profile entrypoints добавляются в operator,
2. новые access правила живут в access,
3. новые payload blocks живут в view/model,
4. `welcome-bot.js` не получает назад profile бизнес-логику.

### 4.3. Расширить Контрактные Тесты
1. Отдельные tests на ACL edge cases.
2. Отдельные tests на partial data profiles.
3. Tests на media/avatar payload уже нужны как постоянный контракт и уже должны расширяться под partial-data кейсы.
4. Tests на live routing invariants: slash, message, open, nav.

## Фаза 5. Релиз И Наблюдаемость

### 5.1. Перед Деплоем
1. Active suite должен быть зелёным.
2. Full `node --test` допускает только известные quarantine leftovers, пока они не очищены.
3. Нужно отдельно зафиксировать manual smoke checklist и прогнать её после рестарта.

### 5.2. После Деплоя
1. Проверить `/профиль` на живом сервере.
2. Проверить helper-message flow.
3. Проверить access deny paths.
4. Проверить navigation update path.
5. Проверить 2-3 реальных профиля с разными состояниями данных.

### 5.3. Что Считать Критическим Багом После Релиза
1. Публичная утечка профиля в канал.
2. `dead`-участник, который может открыть профиль.
3. Untagged пользователь, который видит чужой профиль.
4. Несовпадение target user между helper/open/nav flows.
5. Профиль, который ломает interaction update chain.

## Что Делать Не Нужно

1. Не делать второй профильный UI параллельно текущему V2 path.
2. Не возвращать orchestration обратно в [welcome-bot.js](welcome-bot.js).
3. Не смешивать privacy UX с публичными сообщениями в канале.
4. Не добавлять декоративные блоки раньше, чем будут честно заполнены ключевые данные.
5. Не маскировать отсутствие данных фальшивыми значениями.

## Практический Следующий Порядок Работ

Если двигаться строго по вашему приоритету, а не по абстрактной инженерной красоте, то следующий порядок такой:

1. Пройти Discord smoke на текущем приватном profile V1.
2. Проверить живую полноту данных по 3-5 реальным профилям.
3. Добить data-rich presentation под ваши явные требования: kills history, ranks, Roblox/social, mains/wiki, avatars/media.
4. После этого уже дочищать legacy profile/my-card остатки и делать финальную polish-итерацию.

## Критерий, Что Профиль Можно Считать "Тем Самым"

Профиль можно считать соответствующим вашему видению, когда одновременно выполнено всё ниже:

1. `/профиль` и текстовый trigger работают одинаково надёжно.
2. Чужой профиль открывается только по вашим правилам доступа.
3. Весь основной опыт живёт в одном приватном окне.
4. Игрок сразу видит краткую верхнюю сводку и может перейти в подробные разделы.
5. В профиле реально есть серверная аналитика, progression, Roblox/social и ranking-контекст, а не только заглушки.
6. Интерфейс выглядит как сильный продуктовый экран, а не как набор технических строк.
7. Код не завязан обратно на спагетти в [welcome-bot.js](welcome-bot.js).
