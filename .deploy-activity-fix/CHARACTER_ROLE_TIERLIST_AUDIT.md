# Аудит: роли персонажей, онбординг и текстовый тир-лист

Дата: 03.05.2026

Цель этого файла: зафиксировать в одном месте, с какими проблемами пользователь пришёл, что именно удалось установить по коду, какие архитектурные дефекты есть в системе, какие ограничения мешали анализу, что уже менялось в ходе этой сессии, что остаётся рискованным, и что глобально нужно исправлять дальше.

Этот документ не является changelog-коммитом. Это техническая handoff-записка по состоянию системы и по ходу расследования.

## 1. С чем пользователь пришёл

Основная боль пользователя состояла не в одном баге, а в цепочке связанных дефектов:

1. В текстовом тир-листе пропадали или не отображались кластеры персонажей.
2. В блоке персонажей текстового тир-листа отображались некорректные данные, а затем блок вообще исчез.
3. Бот начал использовать не те роли персонажей: вместо старых реально используемых серверных ролей он стал опираться на мусорные или дублирующие роли.
4. Эти мусорные роли начали просачиваться в онбординг и выбор мейнов.
5. Слой связи между JJS-персонажами, ролями Discord и legacy tierlist оказался повреждён.
6. Пользователь отдельно просил не удалять мусорные роли автоматически: их должен выпиливать другой агент. От текущей задачи требовалось именно восстановить правильную архитектуру и правильную привязку канонических ролей.

Итоговая формулировка задачи по смыслу была такой:

- вернуть старые реальные роли персонажей в архитектуру как источник правды;
- перестать считать мусорные роли управляемыми;
- заново связать роли персонажей с текстовым тир-листом и legacy tierlist;
- восстановить блок персонажей в текстовом тир-листе;
- брать оценку персонажа из legacy tierlist без разрушения существующего сервера.

## 2. Что представляет собой система в целом

### 2.1. Точка входа

Главный entrypoint проекта — welcome-bot.js.

Этот файл перегружен обязанностями. В нём одновременно находятся:

- запуск Discord-клиента;
- welcome/onboarding flow;
- логика назначения ролей;
- статистика и сбор текстового тир-листа;
- интеграция с legacy ELO;
- интеграция с legacy tierlist;
- панели модератора;
- refresh оркестрация;
- часть инфраструктурной логики по данным, ролям и сообщениям.

По факту welcome-bot.js — это одновременно controller, service-layer и часть data-layer.

### 2.2. Источники конфигурации

Система тянет конфигурацию сразу из нескольких слоёв:

1. bot.config.json — статическая конфигурация из репо.
2. env vars — runtime overrides.
3. appConfig — итоговая runtime-конфигурация, собранная из fileConfig + env.
4. db.config из welcome-db.json — динамически накопленное состояние бота.

Это уже создаёт несколько конкурирующих источников правды.

### 2.3. Источники состояния

Основные runtime-данные бота:

1. db.profiles — профили пользователей.
2. db.submissions — заявки пользователей.
3. db.config.generatedRoles — сохранённые маппинги автоматически или полуавтоматически найденных Discord-ролей.
4. db.config.integrations.tierlist — состояние legacy tierlist integration.
5. live legacy tierlist state — внешний state.json, читаемый отдельно от db.

### 2.4. Подсистемы, которые конфликтуют друг с другом

Здесь критично то, что подсистема ролей персонажей и подсистема тир-листа исторически разошлись:

1. Онбординг и выдача ролей смотрят на runtime character entries.
2. Text tierlist stats считают игроков по role-backed character entries.
3. Legacy tierlist sync резолвит mainIds через отдельный resolver.
4. Custom/legacy additions живут в одном месте, а реальные персонажи сервера должны жить в другом.
5. В коде эти границы долгое время были размыты.

### 2.5. Ключевые файлы и их роль

Ниже список файлов, которые оказались центральными для расследования:

1. welcome-bot.js
	Главный монолит. Здесь живут onboarding, role sync, managed role reconcile, text tierlist, legacy tierlist refresh, modal/button handlers и часть data orchestration.
2. src/integrations/character-role-catalog.js
	Вынесенный helper для нормализации managed character catalog, сборки managed entries и historical fallback по role ids.
3. src/integrations/tierlist-character-sync.js
	Resolver-слой между runtime персонажами и legacy tierlist characters. Здесь же helper для cluster lookup и main-id resolution.
4. src/integrations/shared-profile.js
	Нормализация профиля. Критичен тем, что отдельно нормализует mainCharacterIds и characterRoleIds, что влияет на надёжность historical reconstruction.
5. src/onboard/tierlist-stats.js
	Чистый stats-layer для character stats. Важен как downstream-потребитель уже отфильтрованных role-backed entries.
6. tests/tierlist-character-sync.test.js
	Узкие регрессии вокруг runtime-to-legacy sync и cluster semantics.
7. tests/character-role-catalog.test.js
	Узкие регрессии по managed role precedence и historical recovery.

## 3. Как данные должны течь в здоровой архитектуре

Нормальная целевая схема должна быть такой:

1. Есть канонический список JJS-персонажей.
2. Для каждого канонического персонажа есть одна серверная Discord-роль, которую реально используют участники.
3. Онбординг показывает только эти канонические персонажи.
4. syncManagedCharacterRoles снимает и выдаёт только эти канонические роли.
5. getLiveCharacterStatsContext считает статистику только по этим каноническим ролям.
6. buildCharactersRankingEmbed использует эту статистику и накладывает сверху cluster/rating из legacy tierlist.
7. resolveLegacyTierlistMainIdsFromMember сопоставляет те же самые канонические runtime entries с legacy character ids.
8. Custom characters из legacy tierlist не лезут обратно в онбординг и role architecture.

Именно это состояние мы и пытались восстановить.

## 4. Где система была сломана

### 4.1. Смешение канонических персонажей и runtime-добавлений

Одна из центральных проблем — в проекте долго смешивались два разных домена:

1. канонические JJS-персонажи для онбординга и ролей;
2. дополнительные персонажи legacy tierlist/custom characters.

Когда эти домены смешиваются, происходят следующие вещи:

- мусорные персонажи попадают в picker;
- мусорные role ids начинают считаться управляемыми;
- bot start/reconcile начинает работать не с тем набором сущностей;
- text tierlist начинает опираться на повреждённый runtime catalog.

### 4.2. Фрагментированный source of truth для ролей персонажей

На момент анализа роль персонажа могла быть восстановлена или прочитана сразу из нескольких мест:

1. appConfig.characters[].roleId
2. fileConfig.characters[].roleId
3. db.config.generatedRoles.characters[characterId]
4. profile.characterRoleIds
5. submission.mainRoleIds
6. guild.roles cache по имени

Это уже само по себе плохая архитектура.

Проблема не только в количестве источников, но и в том, что между ними не было явно установленного приоритета и явной миграции между состояниями.

### 4.3. Резолв роли по имени слишком хрупок

Часть системы опиралась на предположение, что имя роли Discord совпадает с label персонажа.

На реальном сервере это слишком слабое предположение. Роль могла:

- быть переименована;
- быть создана давно с другим написанием;
- существовать в дублирующем варианте;
- иметь русское или альтернативное имя;
- существовать только по исторически сохранённому ID, но не совпадать по текущему label.

В результате code-path по exact role-name match мог не найти правильную роль и принимать разрушительные решения.

### 4.4. Повреждение блока персонажей в текстовом тир-листе

Текстовый тир-лист зависит от того, что getLiveCharacterStatsContext получает хотя бы один role-backed managed character entry.

Если roleIds оказываются пустыми или стираются, происходит следующее:

1. getCharacterEntries возвращает entries без roleId.
2. getLiveCharacterStatsContext фильтрует их и видит пустой список.
3. characterStats становится пустым.
4. buildCharactersRankingEmbed возвращает null.
5. Блок JJS-персонажей полностью исчезает из текстового тир-листа.

Это и произошло в середине текущей сессии.

### 4.5. Custom tierlist additions протекали обратно в role architecture

submitAddCharacterUnified раньше не просто добавлял custom character в legacy tierlist, но и проталкивал его обратно в runtime character config / role binding flow.

Это архитектурно плохо, потому что:

- custom tierlist additions не обязаны быть онбординг-персонажами;
- у них не обязано быть server-backed canonical role;
- они не должны попадать в picker автоматически.

Это направление потока данных пришлось разрывать.

## 5. Что мешало анализу

Проблемы были не только в бизнес-логике, но и в самой анализируемости системы.

### 5.1. Один огромный монолитный файл

welcome-bot.js слишком большой и смешивает слишком много обязанностей.

Это делает трудными:

- локализацию источника truth;
- быстрый code review;
- поиск побочных эффектов;
- безопасный рефакторинг;
- добавление test seams.

Практический эффект: почти каждая правка требует перечитывать соседние области, потому что рядом может оказаться скрытая зависимость или silent side effect.

### 5.2. Расхождение между workspace и running bot

Во время работы обнаружилось, что:

1. локальный checkout;
2. GitHub VFS workspace;
3. running bot / production state

не всегда отражали одну и ту же ревизию.

Из-за этого часть симптомов в Discord не совпадала с тем, что было видно в текущем коде. Приходилось постоянно перепроверять, тот ли checkout вообще исследуется.

### 5.3. Внешние данные находятся вне репозитория

Ключевые runtime-файлы не живут в репо:

- welcome-db.json
- legacy tierlist state.json
- реальные guild roles

Без них анализ кода даёт только половину картины.

Нам не хватает:

1. актуального welcome-db.json с profiles/submissions/config;
2. фактического state.json legacy tierlist;
3. инвентаризации ролей гильдии;
4. точного понимания, какие роли на сервере исторически считались «правильными».

### 5.4. Часть документации и заметок устарела

Во время анализа обнаружились несоответствия между текущим кодом и ранними заметками о системе.

Например, некоторые старые записи предполагали одну семантику автосоздания/cleanup ролей, а локальный код уже находился в промежуточном состоянии после нескольких итераций фиксов.

### 5.5. Инструментальные трудности

При анализе регулярно мешали и чисто технические вещи:

- grep_search периодически timeout-ился на больших файлах;
- приходилось опираться на read_file и прямые точечные reads вместо широких поисков;
- get_changed_files не всегда был полезен для картирования текущего состояния;
- часть предыдущих промежуточных выводов приходилось перепроверять по живому коду, а не доверять устным summary.

## 6. Что уже было сделано в этой сессии

Ниже перечислено не всё подряд, а только то, что важно для понимания текущего состояния системы.

### 6.1. По линии text tierlist и clusters

1. Был найден и исправлен voted-only cluster filter для row-level labels: canonical zero-vote characters должны сохранять bucket tier label.
2. Логика cluster lookup была вынесена в helper-level слой, чтобы её можно было покрывать тестами.

### 6.2. По линии canonical character roles

1. Был введён отдельный helper для managed character catalog / entries.
2. Character picker перестал брать тупо merged runtime catalog.
3. Custom tierlist additions были частично отсечены от onboarding role architecture.
4. Автоматическое destructive удаление duplicate/orphan character roles было отключено.

### 6.3. По линии регрессии с пропавшим JJS-блоком

После одной из правок выяснилось, что JJS-блок исчез полностью. Это был мой регресс.

Причина:

- reconcile начал очищать generatedRoles.characters, если не находил exact role-name match;
- после этого getLiveCharacterStatsContext видел 0 role-backed персонажей;
- buildCharactersRankingEmbed переставал строиться.

Это было исправлено так:

1. в managed character role helper добавлен historical fallback;
2. добавлено восстановление historical role ids из profiles и submissions;
3. destructive clearing mapping при role-name no-match было убрано;
4. код повторно прогнан через node --check, узкий test и полный npm test.

На момент составления этого файла последний полный прогон был зелёным: 86/86 тестов.

### 6.4. Что дополнительно перепроверено перед созданием этого файла

Перед фиксацией этой записки повторно были просмотрены следующие точки:

1. Нормализация профиля в shared-profile: подтверждено, что mainCharacterIds и characterRoleIds нормализуются независимо.
2. buildHistoricalManagedCharacterRoleIds в character-role-catalog: подтверждено, что историческое восстановление действительно опирается на позиционное соответствие массивов.
3. getLiveCharacterStatsContext в welcome-bot.js: подтверждено, что пустой roleId полностью выбрасывает персонажа из live stats.
4. buildCharactersRankingEmbed в welcome-bot.js: подтверждено, что исчезновение characterStats ведёт к исчезновению всего JJS-блока.
5. reconcileCharacterRolesFromGuild и ensureManagedRoles: подтверждено, что сейчас destructive cleanup отключён, но система всё ещё зависит от слабых binding-эвристик.
6. submitAddCharacterUnified: подтверждено, что custom character теперь должен оставаться в legacy tierlist domain и не должен автоматически становиться onboarding-role сущностью.

## 7. Что сейчас всё ещё плохо архитектурно

Это важный блок. Здесь перечислены не уже исправленные симптомы, а именно сохраняющиеся системные дефекты.

### 7.1. Historical fallback держится на слабом инварианте

Сейчас historical binding строится через buildHistoricalManagedCharacterRoleIds.

Он сопоставляет:

- profile.mainCharacterIds с profile.characterRoleIds;
- submission.mainCharacterIds с submission.mainRoleIds.

Проблема в том, что в shared-profile normalization mainCharacterIds и characterRoleIds нормализуются независимо.

Это означает, что позиционное выравнивание массивов нельзя считать абсолютно надёжным на любой старой или загрязнённой базе.

Если в профиле массивы уже были испорчены до нормализации, то historical reconstruction может восстановить неверный characterId -> roleId.

### 7.2. Слишком много косвенных слоёв до реальной guild role

Вместо одного явного маппинга `characterId -> guildRoleId` система продолжает идти через цепочку:

- configured roleId;
- historical roleId;
- generated roleId;
- exact guild fetch;
- exact role-name match.

Это лучше, чем просто role-name only, но всё ещё плохо как архитектура.

Нужна явная каноническая таблица соответствий, а не вывод по косвенным сигналам.

### 7.3. Picker и role sync всё ещё не валидируют существование роли заранее

Сейчас для entry достаточно иметь непустой roleId, чтобы попасть в role-backed поток.

Это значит, что возможны phantom entries:

- роль формально указана;
- реально на guild её уже нет;
- entry виден и участвует в stats/picker;
- add/remove в syncManagedCharacterRoles тихо проглатывает ошибку.

Это user-facing риск.

### 7.4. Ошибки role assignment проглатываются

В syncManagedCharacterRoles операции add/remove завёрнуты в `.catch(() => {})`.

Это означает, что пользователь и оператор могут видеть, что персонаж как будто выбран, но реальная Discord-роль могла не выдаться.

Такой silent failure особенно плох именно в системах, где source of truth для roles и так нестабилен.

### 7.5. Welcome-bot.js всё ещё слишком много знает

Даже после частичного выноса helper-логики основная архитектурная проблема осталась:

- welcome-bot.js знает про managed characters;
- знает про historical recovery;
- знает про tierlist sync;
- знает про UI;
- знает про cleanup;
- знает про role reconcile.

Это делает систему хрупкой: каждая правка в одной ветке легко ломает соседнюю.

## 8. Что нам не хватает для действительно надёжного решения

### 8.1. Настоящий production state

Нужны реальные данные, а не только код:

1. актуальный welcome-db.json;
2. актуальный legacy state.json;
3. список реальных guild roles;
4. список ролей, которые пользователь считает каноническими исторически.

Без этого часть fixes остаётся эвристикой.

### 8.2. Явный canonical mapping

Нужно отдельное место, где хранится подтверждённое соответствие:

- characterId
- canonical guildRoleId
- display label
- source / verifiedAt

Пока этого нет, бот вынужден угадывать.

### 8.3. Инструментальная диагностика для модератора

Сейчас оператору не хватает нормального health-report по character role architecture.

Нужен минимум один диагностический режим, который показывает:

1. какие canonical characters unresolved;
2. у каких characters roleId ссылается на несуществующую guild role;
3. где найдено несколько кандидатов;
4. какой источник был выбран: configured / historical / generated / name-match;
5. какие entries попадают в picker;
6. какие entries реально считаются в text tierlist.

### 8.4. Деплойная прозрачность

В ходе сессии уже всплывало, что локальная правка не равна обновлённому running bot.

Нужен формализованный минимальный rollout path:

1. локальный fix;
2. tests;
3. deploy;
4. restart / refresh;
5. smoke-check в Discord.

Сейчас это знание размыто между чатом, памятью и отдельными заметками.

## 9. Что мы глобально пытаемся исправить

Если отбросить частные баги, вся эта работа сводится к одной большой цели:

восстановить единый и неразрушительный source of truth для JJS-персонажей и их Discord-ролей, чтобы онбординг, live role sync, text tierlist и legacy tierlist больше не жили в разных реальностях.

Внутри этой цели есть несколько обязательных подцелей:

1. Канонические JJS-персонажи должны быть отделены от custom/legacy additions.
2. Канонические роли не должны удаляться или переопределяться destructive-эвристикой.
3. Онбординг должен видеть только role-backed канонических персонажей.
4. Text tierlist должен считать только тех персонажей, у которых реально есть валидная role binding.
5. Legacy tierlist resolver должен матчить те же самые runtime entries, что и онбординг.
6. Custom tierlist additions не должны обратно засорять role architecture.

Это и есть глобальный архитектурный смысл всей текущей серии фиксов.

## 10. Что нужно сделать дальше

Ниже перечислены не абстрактные идеи, а практический следующий backlog.

### 10.1. Срочные шаги

1. Добавить в managed role path явную проверку существования roleId на guild до того, как entry попадает в picker и в role-backed stats.
2. Перестать считать historical fallback абсолютно доверенным. Его лучше использовать только если configured/generated не резолвятся в живую guild role.
3. Добавить нормальный diagnostic output по unresolved / ambiguous character-role bindings.
4. После любого существенного change в role mapping всегда принудительно refresh-ить text tierlist и проверять, что блок персонажей действительно вернулся.

### 10.2. Среднесрочные шаги

1. Вынести managed character role infrastructure из welcome-bot.js в отдельный модуль.
2. Ввести явную persistent canonical mapping table для characterId -> guildRoleId.
3. Убрать зависимость от index-aligned historical arrays как основного механизма восстановления.
4. Развести типы данных по доменам: onboarding characters отдельно, legacy custom characters отдельно.

### 10.3. Операционные шаги

1. Снять production snapshot текущего welcome-db.json.
2. Снять список реальных ролей Discord по персонажам.
3. Зафиксировать список ролей, которые пользователь считает «старыми правильными».
4. После каждого такого шага прогонять Discord smoke-check, а не только unit tests.

## 11. Краткий вывод

Пользователь пришёл с проблемой «тирлист по персонажам развалился и бот потерял нормальные роли». Анализ подтвердил, что это не косметический баг, а следствие глубокой архитектурной путаницы между:

- каноническими персонажами;
- runtime/generated role bindings;
- профилями и сабмитами;
- legacy tierlist;
- custom additions.

За время сессии удалось:

1. локализовать корневые места разлома;
2. вернуть часть связности между role architecture и text tierlist;
3. устранить один серьёзный регресс, из-за которого вообще пропадал JJS-блок;
4. не трогать мусорные роли destructive-операциями;
5. довести код до зелёного состояния по автотестам.

Но система всё ещё нуждается в доработке. Главный незакрытый долг — отсутствие единого, явного и проверяемого source of truth для character-role mapping.

Пока это не сделано, бот остаётся в режиме «восстановления по косвенным следам», а не в режиме действительно здоровой архитектуры.
